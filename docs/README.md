# Witsale 慧售 - 项目信息中心

## 项目概览

**项目名称**：Witsale (慧售)  
**项目类型**：多端企业销售管理系统（混合B2B+B2C模式）  
**核心用户**：企业员工、渠道客户、终端客户、最终客户  
**技术栈**：FastAPI + PostgreSQL + Redis + React + React Native  
**架构模式**：DDD (领域驱动设计) + 知识图谱/Ontology  

---

## 🔒 强制性规则

### 1. 开发流程规则

- **先设计后编码**：所有功能必须先有设计文档，再进行编码
- **分支管理**：使用Git Flow工作流
  - `main`：稳定版本
  - `develop`：开发分支
  - `feature/*`：功能分支
  - `hotfix/*`：紧急修复分支
- **提交规范**：使用约定式提交 (Conventional Commits)
  - `feat`：新功能
  - `fix`：bug修复
  - `docs`：文档更新
  - `style`：代码风格
  - `refactor`：代码重构
  - `test`：测试
  - `chore`：其他

### 2. 代码规范规则

- **后端**：
  - 遵循PEP8规范
  - 类型提示必须使用
  - Docstring必须完整
  - 函数长度不超过50行
  - 类不超过300行

- **前端**：
  - ESLint + Prettier
  - 组件化开发
  - 命名规范：PascalCase组件，camelCase变量
  - 函数长度不超过40行

### 3. 沟通规则

- **会议记录**：每次讨论必须更新会议记录到 `docs/meetings/`
- **决策记录**：重要技术决策必须记录到 ADR (架构决策记录)
- **问题跟踪**：使用项目看板或Issues跟踪问题
- **每日同步**：至少每日一次项目状态更新

### 4. 文档更新规则

- **实时更新**：任何需求变更、技术决策必须立即更新对应文档
- **文档审查**：重要文档必须经过至少一次审查
- **版本化**：文档必须与代码版本保持同步
- **可访问性**：所有文档必须清晰易读，避免技术术语堆砌

### 5. 版本控制规则

- **提交频率**：小批量、高频提交
- **提交信息**：清晰、简洁、描述性
- **合并请求**：必须有代码审查
- **标签管理**：使用语义化版本号 (x.y.z)

### 6. 测试规则

- **单元测试**：覆盖率 > 80%
- **集成测试**：关键流程必须覆盖
- **E2E测试**：核心业务流程必须测试
- **性能测试**：关键接口必须压测

### 7. 安全规则

- **代码审查**：安全相关代码必须双人审查
- **敏感信息**：禁止硬编码敏感信息
- **依赖管理**：定期更新依赖，避免安全漏洞
- **安全扫描**：定期进行安全扫描
