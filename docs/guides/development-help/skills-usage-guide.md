# 技能使用指南

## 1. 技能概述

项目中目前可用的主要技能是 **Brainstorming 技能**，它是一个用于设计和规范阶段的核心技能，强调在实施前进行充分的设计和用户批准。

- **文件路径**：`/home/lb/mywork/witsale/.trae/brainstorming/SKILL.md`
- **相关文件**：
  - `spec-document-reviewer-prompt.md`：用于审查规范文档的提示
  - `visual-companion.md`：关于视觉辅助功能的文档
  - `scripts` 目录：包含相关脚本

## 2. Brainstorming 技能使用流程

Brainstorming 技能的使用流程分为以下几个关键步骤：

### 2.1 准备阶段

1. **探索项目上下文**
   - 检查项目文件、文档和最近的提交
   - 了解项目的当前状态和结构
   - 识别项目的约束和要求

2. **评估视觉需求**
   - 确定是否需要视觉辅助
   - 如果需要，提供视觉辅助工具

### 2.2 设计阶段

3. **询问澄清问题**
   - 一次一个问题，逐步了解项目需求
   - 优先使用多项选择题，便于用户回答
   - 关注项目的目的、约束和成功标准

4. **提出多种方法**
   - 提出 2-3 种不同的实现方法
   - 分析每种方法的优缺点和权衡
   - 提供推荐方案及其理由

5. **呈现设计**
   - 根据项目复杂度，分部分呈现设计
   - 每部分后获得用户批准
   - 涵盖架构、组件、数据流、错误处理和测试

### 2.3 文档阶段

6. **编写设计文档**
   - 将验证后的设计写入规范文档
   - 保存到 `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`
   - 提交设计文档到版本控制系统

7. **规范审查循环**
   - 调用规范文档审查子代理
   - 如有问题，修复后重新审查
   - 最多循环 5 次，之后需要人工指导

8. **用户审查**
   - 请用户审查规范文档
   - 根据用户反馈进行修改
   - 获得用户批准后进入实施阶段

### 2.4 实施阶段

9. **过渡到实施**
   - 调用 writing-plans 技能创建详细的实施计划
   - 按照实施计划执行编码工作

## 3. 详细步骤说明

### 3.1 探索项目上下文

- **操作**：检查项目文件、文档和最近的提交
- **目的**：了解项目的当前状态和结构
- **输出**：项目上下文概述

### 3.2 提供视觉辅助

- **操作**：如果需要，提供视觉辅助工具
- **目的**：帮助用户更好地理解视觉相关的设计
- **输出**：视觉辅助工具的使用权限

### 3.3 询问澄清问题

- **操作**：一次一个问题，逐步了解项目需求
- **目的**：确保充分理解项目的目的、约束和成功标准
- **输出**：明确的项目需求和约束

### 3.4 提出多种方法

- **操作**：提出 2-3 种不同的实现方法，分析每种方法的优缺点
- **目的**：找到最佳的实现方案
- **输出**：推荐的实现方案及其理由

### 3.5 呈现设计

- **操作**：分部分呈现设计，每部分后获得用户批准
- **目的**：确保设计符合用户期望
- **输出**：用户批准的设计方案

### 3.6 编写设计文档

- **操作**：将验证后的设计写入规范文档
- **目的**：记录设计决策和规范
- **输出**：完整的设计规范文档

### 3.7 规范审查循环

- **操作**：调用规范文档审查子代理，如有问题，修复后重新审查
- **目的**：确保设计规范的质量
- **输出**：审查通过的设计规范

### 3.8 用户审查

- **操作**：请用户审查规范文档，根据用户反馈进行修改
- **目的**：确保设计规范符合用户期望
- **输出**：用户批准的设计规范

### 3.9 过渡到实施

- **操作**：调用 writing-plans 技能创建详细的实施计划
- **目的**：为编码实施做准备
- **输出**：详细的实施计划

## 4. 最佳实践

### 4.1 核心原则

- **一次一个问题**：不要同时提出多个问题，避免用户不知所措
- **优先使用多项选择题**：多项选择题比开放式问题更容易回答
- **严格遵循 YAGNI 原则**：移除设计中不必要的功能
- **探索替代方案**：在确定方案前，始终提出 2-3 种不同的方法
- **增量验证**：分部分呈现设计，每部分后获得用户批准
- **保持灵活性**：当用户反馈不明确时，及时回去澄清

### 4.2 设计原则

- **隔离和清晰**：将系统分解为较小的单元，每个单元都有明确的目的
- **良好的边界**：确保每个单元可以独立理解和测试
- **遵循现有模式**：在现有代码库中工作时，遵循现有的代码模式
- **有针对性的改进**：在工作过程中，对影响工作的现有代码问题进行有针对性的改进
- **保持专注**：不要提出与当前目标无关的重构

### 4.3 视觉辅助使用原则

- **有针对性地使用**：只为需要视觉解释的问题使用视觉辅助
- **独立消息**：视觉辅助的提供必须是独立的消息，不与其他内容混合
- **每问题决策**：即使用户接受了视觉辅助，也要为每个问题决定是否使用浏览器或终端

## 5. 常见问题

### 5.1 什么时候应该使用 Brainstorming 技能？

- 当你需要创建新功能时
- 当你需要构建新组件时
- 当你需要添加新功能时
- 当你需要修改现有行为时
- 当你需要探索用户意图、需求和设计时

### 5.2 什么时候不应该使用 Brainstorming 技能？

- 当你已经有明确的设计和规范时
- 当你只是执行已批准的实施计划时
- 当你只是进行简单的代码修复时

### 5.3 如何处理大型项目？

- 如果项目太大，无法在单个规范中处理，帮助用户将其分解为子项目
- 识别独立的部分，它们如何关联，以及构建它们的顺序
- 然后通过正常的设计流程对第一个子项目进行头脑风暴
- 每个子项目都有自己的规范 → 计划 → 实施周期

### 5.4 如何处理视觉设计问题？

- 当你预计即将到来的问题将涉及视觉内容时，提供视觉辅助
- 对于视觉内容（如模型、线框图、布局比较、架构图），使用浏览器
- 对于文本内容（如需求问题、概念选择、权衡列表），使用终端

## 6. 流程流程图

```
探索项目上下文 → 视觉问题？ → 提供视觉辅助 → 询问澄清问题 → 提出多种方法 → 呈现设计 → 用户批准？ → 编写设计文档 → 规范审查 → 审查通过？ → 用户审查 → 调用 writing-plans 技能
```

## 7. 注意事项

- **硬门控**：在呈现设计并获得用户批准之前，不要调用任何实施技能、编写任何代码、搭建任何项目或采取任何实施行动
- **简单项目也需要设计**：即使是简单的项目，也需要经过设计流程，以避免未检查的假设导致的工作浪费
- **规范审查循环**：最多循环 5 次，之后需要人工指导
- **用户审查门控**：在开始实施计划之前，必须获得用户对规范文档的批准
- **终端状态**：Brainstorming 技能的终端状态是调用 writing-plans 技能，不要调用其他实施技能

## 8. 总结

Brainstorming 技能是项目开发过程中的重要工具，它通过结构化的流程帮助将想法转化为完整的设计和规范。正确使用这个技能可以确保项目的质量和用户满意度，避免不必要的工作和返工。

遵循本文档中的指南，您可以有效地使用 Brainstorming 技能，为项目的成功做出贡献。
