# Witsale 慧售 - 开发流程规范

## 核心原则

**先沟通，后编码** - 任何代码修改前必须充分沟通，达成一致后再执行。

---

## 开发流程规范

### 1. 沟通阶段

**必须执行**：
- 与相关团队成员（主要是项目负责人）进行充分沟通
- 明确需求背景、技术方案及预期目标
- 讨论可能的风险和影响
- 达成一致意见

**沟通方式**：
- 优先使用当前聊天窗口
- 重要事项必须记录到会议记录
- 技术决策必须记录到ADR

### 2. 实施模式

**达成一致后，必须选择以下两种模式之一**：

#### 2.1 /spec 模式（规格说明模式）

**适用场景**：
- 新功能开发
- 复杂技术实现
- 架构变更
- 涉及多个模块的修改

**执行步骤**：
1. 使用 `/spec` 命令进入规格说明模式
2. 编写详细的规格说明文档
3. 包含：
   - 需求背景
   - 技术方案
   - 架构设计
   - 数据模型
   - API设计
   - 测试计划
4. 提交规格说明文档
5. 等待审批
6. 审批通过后开始编码

#### 2.2 /plan 模式（实施计划模式）

**适用场景**：
- 中等复杂度的功能
- 模块内的改进
- 性能优化
- 技术债务清理

**执行步骤**：
1. 使用 `/plan` 命令进入实施计划模式
2. 制定具体实施计划
3. 包含：
   - 任务分解
   - 时间估计
   - 风险评估
   - 测试策略
4. 提交实施计划
5. 等待审批
6. 审批通过后开始编码

### 3. 例外情况

**仅在以下情况可例外**：
- 紧急简单问题
- 明显错误修复
- 文档更新
- 代码风格调整

**例外流程**：
- 简要说明原因
- 快速实施
- 记录到会议记录

### 4. 文档要求

**必须形成书面文档**：
- 沟通内容
- 决策过程
- 实施细节
- 测试结果
- 部署记录

**文档保存位置**：
- 会议记录：`docs/meetings/`
- 架构决策：`docs/adr/`
- 规格说明：`docs/architecture/`
- 实施计划：`docs/guides/`

### 5. 可追溯性

**确保开发过程可追溯、可审计**：
- 所有代码提交必须关联相关文档
- 文档必须包含时间戳和版本信息
- 重要决策必须有明确的理由
- 变更必须有完整的记录

## 流程执行示例

### 示例1：新功能开发（/spec 模式）

1. **沟通**：讨论新功能需求
2. **进入 /spec 模式**：编写详细规格说明
3. **审批**：等待规格说明审批
4. **编码**：按规格说明实现
5. **测试**：执行测试计划
6. **部署**：部署到测试环境
7. **文档**：更新相关文档

### 示例2：bug修复（/plan 模式）

1. **沟通**：分析bug原因
2. **进入 /plan 模式**：制定修复计划
3. **审批**：等待计划审批
4. **修复**：实施修复方案
5. **测试**：验证修复效果
6. **部署**：部署修复
7. **文档**：记录修复过程

---

## 违规处理

- **轻微违规**：口头提醒，补充文档
- **严重违规**：暂停开发权限，要求补充完整流程
- **重复违规**：重新培训开发流程
