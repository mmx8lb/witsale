# Witsale 慧售 - AI 助手项目指南

## 项目概览
- **名称**: Witsale (慧售)
- **类型**: 多端企业销售管理系统 (B2B+B2C)
- **核心用户**: 企业员工、渠道客户、终端客户、最终客户
- **规模**: 大型复杂项目，包含7个端（4个Web + 3个App）

## 技术栈
- **后端**: FastAPI + PostgreSQL + Redis + DDD + 知识图谱
- **前端**: React + React Native
- **数据库**: PostgreSQL 15+ (JSONB, Array, Range Types)
- **缓存**: Redis 7+
- **知识图谱**: owlready2 + rdflib

## 核心功能模块
1. **用户与认证** (RBAC权限)
2. **商品管理** (多级定价 + 特殊策略)
3. **订单管理** (多级订单)
4. **库存管理** (分布式多级库存)
5. **财务管理** (多种结算方式)
6. **外勤管理** (打卡 + 轨迹)
7. **知识图谱/Ontology** (智能推荐 + 语义搜索)

## 管理端架构
- **内部管理端**: 企业员工使用
- **外部管理端**: 渠道/终端客户使用

## 开发流程规范

### 核心原则
**先沟通，后编码** - 任何代码修改前必须充分沟通，达成一致后再执行。

### 执行步骤

#### 1. 沟通阶段
**必须执行**：
- 与项目负责人进行充分沟通
- 明确需求背景、技术方案及预期目标
- 讨论可能的风险和影响
- 达成一致意见

**沟通方式**：
- 优先使用当前聊天窗口
- 重要事项必须记录到会议记录
- 技术决策必须记录到ADR

#### 2. 实施模式
**达成一致后，必须选择以下两种模式之一**：

##### 2.1 /spec 模式（规格说明模式）
**适用场景**：
- 新功能开发
- 复杂技术实现
- 架构变更
- 涉及多个模块的修改

**执行步骤**：
1. 使用 `/spec` 命令进入规格说明模式
2. 编写详细的规格说明文档
3. 包含：需求背景、技术方案、架构设计、数据模型、API设计、测试计划
4. 提交规格说明文档
5. 等待审批
6. 审批通过后开始编码

##### 2.2 /plan 模式（实施计划模式）
**适用场景**：
- 中等复杂度的功能
- 模块内的改进
- 性能优化
- 技术债务清理

**执行步骤**：
1. 使用 `/plan` 命令进入实施计划模式
2. 制定具体实施计划
3. 包含：任务分解、时间估计、风险评估、测试策略
4. 提交实施计划
5. 等待审批
6. 审批通过后开始编码

#### 3. 例外情况
**仅在以下情况可例外**：
- 紧急简单问题
- 明显错误修复
- 文档更新
- 代码风格调整

**例外流程**：
- 简要说明原因
- 快速实施
- 记录到会议记录

#### 4. 文档要求
**必须形成书面文档**：
- 沟通内容
- 决策过程
- 实施细节
- 测试结果
- 部署记录

**文档保存位置**：
- 会议记录：`docs/meetings/`
- 架构决策：`docs/adr/`
- 规格说明：`docs/architecture/`
- 实施计划：`docs/guides/`

#### 5. 可追溯性
**确保开发过程可追溯、可审计**：
- 所有代码提交必须关联相关文档
- 文档必须包含时间戳和版本信息
- 重要决策必须有明确的理由
- 变更必须有完整的记录
