# Witsale 慧售 - AI 助手项目指南

## 项目概览

- **名称**: Witsale (慧售)
- **类型**: 企业销售管理系统核心后台
- **核心用户**: 企业员工
- **规模**: 中型项目，专注于核心后台管理功能

## 技术栈

- **后端**: FastAPI + PostgreSQL + Redis + DDD + 知识图谱
- **前端**: React + Ant Design (核心后台管理系统)
- **数据库**: PostgreSQL 15+ (JSONB, Array, Range Types)
- **缓存**: Redis 7+
- **知识图谱**: owlready2 + rdflib

## 核心功能模块

1. **用户与认证** (RBAC权限)
2. **商品管理** (多级定价 + 特殊策略)
3. **订单管理** (多级订单)
4. **库存管理** (分布式多级库存)
5. **财务管理** (多种结算方式)
6. **外勤管理** (打卡 + 轨迹)
7. **知识图谱/Ontology** (智能推荐 + 语义搜索)

## 管理端架构

- **核心后台管理系统**: 企业员工使用，包含所有核心功能模块

## 开发流程规范

### 核心原则

**先沟通，后编码** - 任何代码修改前必须充分沟通，达成一致后再执行。

### 执行步骤

#### 1. 沟通阶段

**必须执行**：

- 与项目负责人进行充分沟通
- 明确需求背景、技术方案及预期目标
- 讨论可能的风险和影响
- 达成一致意见

**沟通方式**：

- 优先使用当前聊天窗口
- 重要事项必须记录到会议记录
- 技术决策必须记录到ADR

#### 2. 实施模式

**达成一致后，必须选择以下两种模式之一**：

##### 2.1 /spec 模式（规格说明模式）

**适用场景**：

- 新功能开发
- 复杂技术实现
- 架构变更
- 涉及多个模块的修改

**执行步骤**：

1. 使用 `/spec` 命令进入规格说明模式
2. 编写详细的规格说明文档
3. 包含：需求背景、技术方案、架构设计、数据模型、API设计、测试计划
4. 提交规格说明文档
5. 等待审批
6. 审批通过后开始编码

##### 2.2 /plan 模式（实施计划模式）

**适用场景**：

- 中等复杂度的功能
- 模块内的改进
- 性能优化
- 技术债务清理

**执行步骤**：

1. 使用 `/plan` 命令进入实施计划模式
2. 制定具体实施计划
3. 包含：任务分解、时间估计、风险评估、测试策略
4. 提交实施计划
5. 等待审批
6. 审批通过后开始编码

#### 3. 例外情况

**仅在以下情况可例外**：

- 紧急简单问题
- 明显错误修复
- 文档更新
- 代码风格调整

**例外流程**：

- 简要说明原因
- 快速实施
- 记录到会议记录

#### 4. 文档要求

**必须形成书面文档**：

- 沟通内容
- 决策过程
- 实施细节
- 测试结果
- 部署记录

**文档保存位置**：

- 会议记录：`docs/meetings/`
- 架构决策：`docs/adr/`
- 规格说明：`docs/architecture/`
- 实施计划：`docs/guides/`

#### 5. 可追溯性

**确保开发过程可追溯、可审计**：

- 所有代码提交必须关联相关文档
- 文档必须包含时间戳和版本信息
- 重要决策必须有明确的理由
- 变更必须有完整的记录

## AI 助手开发规范

### 核心原则

**安全第一** - 任何操作前必须确认安全性，避免误操作导致项目损坏。

### 行为约束

1. **禁止重建项目**：除非明确指示，否则不得重建整个项目结构
2. **禁止删除文件**：除非明确指示，否则不得删除现有文件
3. **禁止修改核心配置**：除非明确指示，否则不得修改数据库连接、API路由等核心配置
4. **备份原则**：在修改重要文件前，应先查看文件内容并确认修改的必要性
5. **验证原则**：修改后应验证修改是否正确，确保系统正常运行
6. **文档优先**：遵循项目的文档结构，确保所有开发活动都有相应的文档记录
7. **测试优先**：任何功能修改后，应运行相关测试确保功能正常
8. **沟通优先**：遇到不确定的情况，应先沟通确认后再执行

### 开发流程

1. **分析现状**：了解项目当前状态和结构
2. **制定计划**：根据需求制定详细的开发计划
3. **执行实施**：按照计划逐步实施，确保每一步都有验证
4. **测试验证**：完成开发后进行全面测试
5. **文档更新**：更新相关文档，确保可追溯性

### 紧急情况处理

1. **发现错误**：如发现代码错误，应立即停止操作并报告
2. **系统异常**：如系统出现异常，应立即停止操作并分析原因
3. **数据安全**：确保数据安全，避免数据丢失或损坏

### 最佳实践

1. **代码风格**：严格遵循项目的代码风格规范
2. **版本控制**：定期提交代码，确保版本控制的完整性
3. **注释说明**：为复杂代码添加清晰的注释说明
4. **性能考虑**：考虑代码的性能影响，避免性能瓶颈
5. **安全考虑**：考虑代码的安全性，避免安全漏洞

### 沟通机制

1. **定期汇报**：定期汇报开发进度和遇到的问题
2. **问题记录**：记录开发过程中遇到的问题和解决方案
3. **知识共享**：分享开发过程中的经验和知识

## AI与开发者双人协作流程

### 流程引用

**核心协作规则**：所有开发活动必须遵循 `docs/rules/ai-collaboration-rules.md` 中定义的AI与开发者协作规则。

### 流程概述

1. **代码分析阶段**：AI对现有代码进行全面分析，识别潜在问题、优化点及改进机会
2. **方案审查阶段**：开发者审查AI提出的改进方案并明确批准
3. **编码实现阶段**：AI在获得批准后执行编码实现
4. **验证与迭代阶段**：开发者验证实现结果，AI分析更新后的代码，进入下一个迭代

### 执行要求

- **必须遵循**：每次开发任务都应按照协作规则执行
- **文档记录**：所有分析、方案和实施过程都应记录在相应文档中
- **审批机制**：任何代码修改必须获得开发者的明确批准
- **质量保证**：确保代码质量和功能正确性符合预期
